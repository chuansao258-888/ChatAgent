# MQ 异步架构 Stage 4C+ 补丁落地清单 (V2.5)

## 1. 背景与目标
在针对 Stage 4A/4B/4C 的对抗性评审中，识别出了分布式环境下的关键缺陷：特别是 **RUNNING 锁黑洞导致的异步任务丢失**、分布式 SSE 消息丢失、以及重试幂等不足。本计划旨在加固架构底座，确保异步链路的“At-least-once”保证与高可用性。

---

## 2. 核心修补任务清单

### 任务 1：分布式 SSE 消息分发 (架构阻断项 - P0)
**目标**：解决多节点部署下异步消息触达问题。
- **1.1 接口拆分**：`SseService` 拆分 `publish` (广播) 与 `deliverLocal` (本地投递)。
- **1.2 Redis 广播**：发布端推 Redis Channel，订阅端命中本地 Emitter 后执行投递。
- **1.3 序列化固化**：确保所有 SSE Payload 具备标准 JSON 序列化能力。

### 任务 2：重试幂等与“手术级”回滚 (逻辑加固项 - P1)
**目标**：消除重试/重放导致的重复输出。
- **2.1 针对性清理**：实现 `rollbackTurn(sessionId, turnId)`，清理 `ASSISTANT` 和 `TOOL` 消息。
- **2.2 有条件触发**：仅在重试/重放或未完成遗留场景下触发，严防误伤正常数据。

### 任务 3：会话级并发控制 (分阶段治理 - P2)
**目标**：降低状态竞争概率。
- **3.1 UI 侧硬拦截**：`isProcessing` 时禁用发送。标注为“交互层缓解”，不提供服务端最终一致性保证。
- **3.2 后端串行化**：(暂缓) 视后期压力情况引入基于 Redis 的 Session 锁。

### 任务 4：感知延迟优化 (UX 改进项 - P1)
**目标**：消除异步链路感知空窗。
- **4.1 缩短轮询**：Outbox Poller 间隔调优至 **500ms**。
- **4.2 前端本地 Pending**：UI 立即显示占位 Loading，确保状态真实性。

### 任务 5：Stage 4A/4B 可靠性加固 (架构底座补丁 - P0/P2)
**目标**：修复评审中发现的“锁黑洞”等严重可靠性漏洞。
- **5.1 修复 RUNNING 锁导致的任务丢失 (P0)**
    - **Outcome 细化**：`DistributedLockManager.tryAcquire` 需区分 `RUNNING` 和 `COMPLETED`。
    - **退避策略**：`AbstractRetryingMqConsumer` 若遇到他人持有的 `RUNNING` 锁，必须执行 **`nack(true)` (重入队)**，严禁 `ack` 跳过。
    - **风险对策**：意识到 `nack(true)` 存在原地重入队的热循环风险。此项作为**最小安全修复**落地，后续应配套 RabbitMQ `prefetch` 调优或增加后端主动回退逻辑。
- **5.2 Outbox 幽灵重发防御 (P2)**
    - **设计约束**：承认 Outbox 在极低概率下会出现重复发送（Stale Claim 窗口）。要求业务侧必须利用 `x-idempotency-key` 实现强幂等。
- **5.3 Redis 故障降级决策 (P2)**
    - **精细化降级**：定义 Redis 宕机时的行为。支持 **按任务类型 (Task Type) 分别配置 Fail-fast 或 Fail-open**（例如：`ingest.task` 倾向一致性选 Fail-fast；`agent.run` 倾向可用性选 Fail-open）。

---

## 3. 落地优先级建议

| 优先级 | 阶段 | 核心行动项 |
| :--- | :--- | :--- |
| **P0 (Critical)** | **架构与可靠性** | **RUNNING 锁黑洞修复 (5.1)** & 分布式 SSE 分发 (1.1, 1.2) |
| **P1 (High)** | **逻辑与体验** | 重试回滚机制 (2.1, 2.2) & 感知延迟优化 (4.1, 4.2) |
| **P2 (Medium)** | **UX 与鲁棒性** | UI 发送拦截 (3.1) & 细粒度降级策略固化 (5.2, 5.3) |

---

## 4. 验收标准
- **可靠性验证**：人为模拟消费者宕机，观察消息重投递后是否被 `nack` 并由其他节点成功消费（而非被 Ack 吞掉）。
- **分布式验证**：双节点环境下，异步生成的 Token 必须能跨节点推送到 SSE 客户端。
- **数据一致性验证**：重试结束后，数据库同一 `sessionId` + `turnId` 下仅保留最新一次完整输出集合。
- **感知验证**：提问后 500ms 内 UI 进入 Pending 态，且处理中状态无法重复提问。
